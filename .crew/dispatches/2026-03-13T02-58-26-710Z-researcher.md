## Research: Rubrics in LLM-as-Judge Evaluation Systems

### Answer

A **rubric** in LLM evaluation is a structured definition of how an LLM judge should grade outputs. It combines:
1. A **prompt template** (instructions for the LLM judge)
2. **Choice definitions** (discrete options the judge can select)
3. **Score mappings** (how each choice maps to a numeric score between 0–1)
4. Optional **choice_scores** that define different point values for different answers

The rubric, along with the test case input and expected output, is passed to the LLM judge via a templated prompt. The judge selects one choice, which is automatically converted to a numeric score via the rubric's choice_scores mapping.

### Key Findings

#### 1. **What is a Rubric: Definition & Purpose**

A rubric is a **scoring specification** that defines:
- **Prompt**: The instruction text the LLM judge receives (with template variables like `{{input}}`, `{{output}}`, `{{expected}}`)
- **Choice strings**: Discrete options the judge must pick from (e.g., A, B, C, D, E)
- **Choice scores**: A mapping from each choice to a numeric score (0–1 range)

Example (Braintrust Autoevals Factuality rubric):
```yaml
prompt: |
  Compare the factual content of the submitted answer with the expert answer...
  (A) The submitted answer is a subset...
  (B) The submitted answer is a superset...
  ...
choice_scores:
  "A": 0.4
  "B": 0.6
  "C": 1.0
  "D": 0.0
  "E": 1.0
```

(Source: https://raw.githubusercontent.com/braintrustdata/autoevals/main/templates/factuality.yaml)

#### 2. **Frameworks Define Rubrics via Structured Data Models**

**OpenAI Evals** uses a `ModelGradedSpec` dataclass:
```python
@dataclass
class ModelGradedSpec:
    prompt: Union[str, OpenAICreateChatPrompt]          # The rubric prompt
    choice_strings: Union[list[str], str]              # Available choices
    input_outputs: dict[str, str]                      # Field mappings
    choice_scores: Optional[dict[str, float]] = None   # Choice → score mapping
    eval_type: Optional[str] = None
```

(Source: https://raw.githubusercontent.com/openai/evals/main/evals/elsuite/modelgraded/base.py)

**Braintrust Autoevals** uses an `LLMClassifier` with:
- `name`: Evaluator identifier
- `prompt_template`: Mustache-style template with `{{output}}`, `{{expected}}`, `{{input}}`
- `choice_scores`: Dict mapping choice strings to scores (0–1)
- `model`: LLM to use as judge
- `use_cot`: Enable chain-of-thought reasoning (default: True)

(Source: https://raw.githubusercontent.com/braintrustdata/autoevals/main/py/autoevals/llm.py)

**Braintrust Evals SDK** uses an `Eval` with `scores` array:
```typescript
Eval("Say Hi Bot", {
  data: () => [{ input, expected }],
  task: (input) => llm_output,
  scores: [LevenshteinScorer, FactualityScorer],  // Array of scorers
});
```

(Source: https://github.com/braintrustdata/braintrust-sdk-javascript)

#### 3. **Rubric Data Model: Structured vs. Unstructured**

Rubrics are **semi-structured**:
- **Unstructured component**: The prompt text is free-form natural language (can include detailed instructions, examples, reasoning guidelines)
- **Structured component**: Choice definitions and scoring are discrete and typed

The prompt can include:
- Context about the evaluation task
- Instructions on what to ignore (style vs. factuality)
- Multi-turn reasoning guidance (chain-of-thought)
- Template variables that get filled at eval time

#### 4. **Evaluation Flow: How Rubric + Input + Output → Verdict**

The evaluation follows this sequence:

1. **Prompt Template Rendering**: Variables (`{{input}}`, `{{output}}`, `{{expected}}`) are substituted into the rubric prompt
2. **LLM Judge Inference**: The judge (e.g., GPT-4) receives:
   - System message (implicit: "you are an evaluator")
   - User message: rendered prompt + choice instructions
   - Tool schema: structured function with `choice` parameter + optional `reasoning` (for chain-of-thought)
3. **Tool Call Parsing**: Judge calls `select_choice(choice="A")` or similar
4. **Score Extraction**: Framework looks up `choice_scores["A"]` → numeric score
5. **Metadata Collection**: Optional `reasoning` field from the judge is stored as metadata
6. **Score Object Creation**: Returns a `Score` object with:
   - `name`: Evaluator name
   - `score`: Float 0–1
   - `metadata`: Dict with choice, rationale, reasoning

Example from Braintrust:
```python
result = Factuality(output="...", expected="...", input="...")
# Returns: Score(name="Factuality", score=1.0, metadata={"choice": "C", "rationale": "..."})
```

(Source: https://raw.githubusercontent.com/braintrustdata/autoevals/main/py/autoevals/score.py)

#### 5. **Key Rubric Design Patterns**

- **Symmetric scoring**: Some choices worth 0 (wrong), 1 (right), others worth 0.4–0.6 (partial)
- **Factuality rubrics**: Distinguish between subsets/supersets/exact matches/disagreements
- **Chain-of-Thought suffix**: Rubrics typically append reasoning instructions to force step-by-step judgment
- **Tool-forced output**: Using OpenAI function calling prevents the judge from writing free text; it must select from the rubric's choices

#### 6. **Bias & Reliability Considerations**

LLM-as-judge evaluation has known limitations (per LMSYS research):
- **Position bias**: Judge favors first/last options
- **Verbosity bias**: Longer outputs scored higher
- **Self-enhancement bias**: Judge favors outputs from its own training distribution
- **Limited reasoning**: Even with chain-of-thought, judges can make mistakes

These are mitigated via:
- Multiple independent judges
- Rubric design (clear choice definitions)
- Comparison mode (A vs. B) vs. absolute scoring

(Source: https://arxiv.org/abs/2306.05685 — "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena")

### Recommended Approach

**For building evaluation systems with rubrics:**

1. **Define rubric as declarative YAML** (like Braintrust/OpenAI Evals):
   ```yaml
   name: factuality_check
   prompt: |
     [Detailed instructions with {{input}}, {{output}}, {{expected}} variables]
   choice_strings: [A, B, C, D]
   choice_scores:
     A: 0.5
     B: 1.0
     C: 0.0
     D: 0.75
   ```

2. **Use structured scorer classes** (e.g., `LLMClassifier` from Autoevals):
   - Handles template rendering
   - Manages tool calling with LLM
   - Parses responses reliably
   - Returns normalized `Score` objects

3. **Compose rubrics into eval workflows**:
   - Map rubrics to test cases (input, expected_output)
   - Run multiple scorers per test case
   - Aggregate scores (mean, min, max, median)
   - Store both score and metadata (rationale) for debugging

4. **Validate rubric quality**:
   - Inter-rater agreement (multiple models)
   - Agreement with human labels (>80% for GPT-4 per LMSYS)
   - Measure position/verbosity bias in pilot evals
