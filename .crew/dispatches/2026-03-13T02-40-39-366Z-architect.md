## Output
## Goal

When this system is complete, a user can manage a collection of test cases (inputs and expected outputs), define rubric-based graders that use an LLM to evaluate those test cases, and run experiments that apply any combination of graders against any dataset — seeing pass/fail verdicts and LLM-generated reasons for each test case × grader pair, all within a single three-tab Next.js interface. State persists for the duration of the session without requiring a backend database.

## Behaviors

- **DatasetRowCreate:** When the user submits a new row in the Dataset tab, that row appears immediately in the table with its `input`, `expected_output`, and any custom field values populated.
- **DatasetRowEdit:** When the user edits a row and confirms the change, the table reflects the updated values immediately without losing any other rows.
- **DatasetRowDelete:** When the user deletes a row, it is removed from the table immediately and is no longer available for any subsequent experiment run.
- **CustomFieldAdd:** When the user adds a custom column to the dataset, every existing and future row displays a cell for that column (empty by default).
- **GraderCreate:** When the user saves a new grader definition, it appears in the Graders list and becomes selectable in the Experiment tab.
- **GraderEdit:** When the user edits a grader's name, description, or rubric and saves, the updated definition is used in all future experiment runs.
- **GraderDelete:** When the user deletes a grader, it is removed from the list and no longer appears as a selectable option in the Experiment tab.
- **ExperimentConfiguration:** When the user opens the Experiment tab, they can select exactly one dataset and one or more graders before running.
- **ExperimentRun:** When the user triggers a run, every test case row is evaluated by every selected grader; the system sends the row's `input`, `expected_output`, and the grader's `rubric` to an LLM and receives a `pass/fail` verdict and a `reason` for each combination.
- **ResultsDisplay:** When a run completes, the results table shows one row per test case and one column per selected grader; each cell contains the verdict (`pass` or `fail`) and the LLM-generated reason.
- **StatePersistence:** When the user switches between tabs, all datasets, grader definitions, and the most recent experiment results are still present exactly as left.
- **RunInProgress:** While a run is executing, the user sees a loading indicator and cannot trigger a second run until the current one completes.

## Contracts

**Test Case (row in a dataset)**
- `id` — unique identifier, system-assigned
- `input` — free text
- `expected_output` — free text
- `custom_fields` — a map of field name to free-text value; may be empty

**Dataset**
- `id` — unique identifier, system-assigned
- `name` — user-provided label
- `rows` — ordered list of Test Cases
- `custom_field_names` — ordered list of strings defining which custom columns exist

**Grader**
- `id` — unique identifier, system-assigned
- `name` — user-provided label
- `description` — free text explaining what the grader checks
- `rubric` — free text instructions sent verbatim to the LLM to guide evaluation

**LLM Evaluation Request (per grader × per test case)**
- Inputs sent: `input`, `expected_output`, `rubric`
- Outputs received: `verdict` (`pass` or `fail`), `reason` (free text from LLM)

**Experiment Result**
- `experiment_id` — unique identifier, system-assigned
- `dataset_id` — reference to the dataset used
- `grader_ids` — list of grader IDs used in this run
- `cells` — list of entries, each with: `row_id`, `grader_id`, `verdict`, `reason`

## Constraints

- Greenfield — no prior implementation.
- The application is a Next.js app; all UI must be compatible with that runtime.
- Grader evaluation must use an LLM via the Vercel AI SDK; no hand-written string-matching logic may substitute for LLM evaluation.
- All state is in-memory for the session; no database reads or writes are required.
- The system must function as a single-user, single-session tool — no authentication, no multi-tenancy.

## Error Cases

- **LLMCallFailure:** An LLM evaluation request fails (network error, timeout, or API error) → the affected cell displays an error state (not `pass` or `fail`) with a human-readable message; all other cells in the run that succeeded are still shown; the user can retry the run.
- **EmptyDatasetRun:** The user attempts to start a run with a dataset that has no rows → the run does not start and the user sees an inline validation message.
- **NoGraderSelected:** The user attempts to start a run without selecting any grader → the run does not start and the user sees an inline validation message.
- **GraderDeletedWhileReferenced:** The user deletes a grader that was used in a previously completed experiment → past results for that grader are still displayed, labeled with the grader's former name (or a tombstone label indicating it was deleted), so result history is not silently corrupted.
- **MalformedLLMResponse:** The LLM returns a response that does not contain a parseable verdict → the affected cell displays an error state; reason text from the LLM (if any) is still surfaced to the user for debugging.

## Out of Scope

- Persisting datasets, graders, or results to any SQL or external database.
- Aggregate statistics such as pass rate per grader or per dataset.
- Exporting results as CSV or any other file format.
- User authentication or multi-user sessions.
- Versioning of datasets or graders.
- Streaming LLM responses into cells as they arrive (cells populate on completion).
- Running multiple experiments concurrently.
- Comparing results across multiple experiment runs.

## Open Questions

1. Should the LLM model (e.g. GPT-4o, Claude 3.5) be hardcoded for the initial version, or does the user need to select or configure it per grader or per experiment?
2. Is there a defined prompt template structure for how `input`, `expected_output`, and `rubric` are assembled before being sent to the LLM, or is the rubric expected to be a fully self-contained prompt that the system passes through as-is?
3. Does the Experiment tab need to support running against multiple saved datasets in a single run, or always exactly one dataset per run?
4. Should custom fields on test case rows be included in the LLM evaluation context (i.e., sent alongside `input` and `expected_output`), or are they metadata only?
5. Is there a concurrency requirement for LLM calls within a single run — should all grader × row evaluations fire in parallel, or must they run sequentially?
6. What is the expected behavior when a grader's rubric is empty — is it a validation error at save time, or is the grader permitted to exist without a rubric?
