# Data Model

## Overview

This document defines the in-memory entity model for the lightweight eval harness. All state lives in a single session-scoped store. No persistence layer is involved — but entities are modeled with the same rigor as if they were database records.

---

## Entities

### 1. `Attribute`

Represents a single column definition in a dataset's schema.

```ts
type Attribute = {
  // System-generated
  id: string                  // Stable local identifier (e.g. UUID or auto-increment)
  insertionOrder: number      // Position within this dataset's schema (0-indexed)
  isBuiltIn: boolean          // true for "input" and "expected_output" only

  // User-provided (or derived)
  name: string                // Always lowercase (per AttributeAdd: auto-lowercased on creation)
}
```

**Constraints:**
- `name` must be unique within the owning dataset's schema.
- `name` must be non-empty.
- `name` is stored lowercase. `"Input"` and `"input"` are the same attribute (per Resolved Decision #2).
- `isBuiltIn` is `true` only for `"input"` and `"expected_output"`. All other attributes have `isBuiltIn = false`.
- The two built-in attributes (`"input"`, `"expected_output"`) are always present and have `insertionOrder` 0 and 1 respectively. Custom attributes start at `insertionOrder = 2`.
- An `Attribute` with `isBuiltIn = true` cannot be removed from its dataset (per BuiltInProtection and RemoveBuiltInAttribute).
- Attributes are ordered by `insertionOrder`. The order is immutable once set (per Resolved Decision #4: no reordering).

**Relationships:**
- Belongs to exactly one `Dataset` (many-to-one).
- Cascade-deleted when its owning `Dataset` is deleted.

### 2. `Dataset`

A named collection of structured test cases, with a fixed schema.

```ts
type Dataset = {
  // System-generated
  id: string                  // Stable local identifier

  // User-provided
  name: string                // Non-empty, unique across all datasets

  // Derived / managed
  schema: Attribute[]         // Ordered list; always contains "input" and "expected_output" first
  items: DatasetItem[]        // Ordered by insertion; no reordering allowed
}
```

**Constraints:**
- `name` must be non-empty (per EmptyDatasetName).
- `name` must be unique across all datasets (per DuplicateDatasetName, Resolved Decision #1).
- `schema` always contains at least two entries: the built-in `"input"` and `"expected_output"` attributes, in that order, from the moment of creation (per DatasetCreate).
- A `Dataset` with zero items does not appear as a selectable option when creating an `Experiment` (per Resolved Decision #6).
- Renaming a `Dataset` preserves all `items` and schema (per DatasetRename).

**Relationships:**
- Has many `Attribute`s (schema). One-to-many. Cascade-deleted on dataset deletion.
- Has many `DatasetItem`s. One-to-many. Cascade-deleted on dataset deletion.
- Referenced by many `Experiment`s. On dataset deletion, all referencing `Experiment`s (and their `ExperimentResultCell`s) are also deleted (per DatasetDelete). User must confirm if any experiments exist (per DatasetDeleteExperimentWarning).

---

### 3. `DatasetItem`

A single row in a dataset. Always conforms to the dataset's current schema.

```ts
type DatasetItem = {
  // System-generated
  id: string                  // Stable local identifier
  insertionOrder: number      // Position within the owning dataset (0-indexed, immutable after creation)

  // Managed
  values: Record<string, string>
  // Keys are exactly the current attribute names of the owning dataset's schema.
  // Values are always strings; missing attributes default to empty string "".
}
```

**Constraints:**
- `values` must contain exactly the keys present in the owning dataset's `schema` at any given moment — no more, no fewer (per ItemSchemaConformance).
- All values are strings. No numeric or structured types (per spec Constraints).
- `"input"` and `"expected_output"` keys are always present in `values`.
- When a new `Attribute` is added to the dataset schema, all existing `DatasetItem`s immediately gain that attribute key with value `""` (per AttributeAdd / ItemSchemaConformance).
- When an `Attribute` is removed from the schema, all existing `DatasetItem`s immediately drop that key (per AttributeRemove / ItemSchemaConformance).
- Items appear in `insertionOrder`. No reordering (per Resolved Decision #3).

**Relationships:**
- Belongs to exactly one `Dataset` (many-to-one).
- Cascade-deleted when its owning `Dataset` is deleted.

---

### 4. `Grader`

An evaluation criterion. Holds the rubric that is given to the LLM judge.

```ts
type Grader = {
  // System-generated
  id: string                  // Stable local identifier

  // User-provided
  name: string                // Non-empty, unique across all graders
  description: string         // May be empty string; never null
  rubric: string              // Non-empty; the full judging instruction text for the LLM
}
```

**Constraints:**
- `name` must be non-empty (per EmptyGraderName).
- `name` must be unique across all graders (per DuplicateGraderName, Resolved Decision #1 for graders).
- `rubric` must be non-empty (per EmptyRubric / EditEmptyRubric).
- `description` is optional and defaults to `""`. An empty description is stored as `""`, never `null` (per Resolved Decision #5).
- No enforced length limit on `rubric` (per Resolved Decision #3 for graders).

**Relationships:**
- Referenced by many `Experiment`s (many-to-many via the `Experiment.graderIds` list).
- On grader deletion, all referencing `Experiment`s (and their `ExperimentResultCell`s) are cascade-deleted. User must confirm if any experiments exist (per GraderDelete, Resolved Decision #4 for graders).

---

### 5. `Experiment`

Ties a dataset and one or more graders together. Tracks run status and owns all result cells.

```ts
type ExperimentStatus = "queued" | "running" | "complete" | "failed"
// - "queued":   created, waiting for another experiment to finish running
// - "running":  actively evaluating cells
// - "complete": all cells have been evaluated (some may be in error state)
// - "failed":   run could not complete or all cells failed (per RunFailureTotal)

type Experiment = {
  // System-generated
  id: string                  // Stable local identifier

  // User-provided
  name: string                // Non-empty; derived from original name on re-run (ExperimentRerun)

  // Relationships (stored as stable IDs)
  datasetId: string           // Reference to the owning Dataset (live reference, not a snapshot)
  graderIds: string[]         // Ordered list of Grader IDs; at least one required

  // System-managed
  status: ExperimentStatus    // Starts as "queued" on creation
  results: ExperimentResultCell[]
}
```

**Constraints:**
- `name` must be non-empty (per MissingRequiredFields).
- `datasetId` must reference an existing `Dataset` that contains at least one item at run time (per EmptyDatasetRun; empty datasets are not selectable at creation, per Resolved Decision #6).
- `graderIds` must contain at least one entry (per MissingRequiredFields).
- `graderIds` has no upper bound (per Resolved Decision #3 for experiments).
- `status` starts as `"queued"` on creation.
- Only one experiment may be in `"running"` state at a time; others queue (per ConcurrentRun, Resolved Decision #5 for experiments).
- The dataset is referenced live — not snapshotted. If the dataset changes after a run, results reference current dataset state (per Resolved Decision #2 for experiments).
- Re-running creates a new `Experiment` entity with a new `id` and derived `name`. The original is preserved (per ExperimentRerun, Resolved Decision #1 for experiments).
- No timestamps are stored (per Resolved Decision #4 for experiments).

**Relationships:**
- References one `Dataset` (many-to-one).
- References one or more `Grader`s (many-to-many, stored as `graderIds[]`).
- Has many `ExperimentResultCell`s. One-to-many. Cascade-deleted when the `Experiment` is deleted (per ExperimentDelete).
- Cascade-deleted when its referenced `Dataset` is deleted (per DatasetDelete).
- Cascade-deleted when any of its referenced `Grader`s is deleted (per GraderDelete).

### 6. `ExperimentResultCell`

One verdict for a single dataset item × grader pair within an experiment.

```ts
type Verdict = "pass" | "fail" | "error"
// - "pass":  LLM judge determined the item output meets the rubric criteria
// - "fail":  LLM judge determined the item output does not meet the rubric criteria
// - "error": The evaluation call failed (per RunFailurePartial); no verdict was produced

type ExperimentResultCell = {
  // System-generated
  id: string                  // Stable local identifier

  // Relationship keys
  experimentId: string        // Owning experiment
  datasetItemId: string       // The dataset item that was evaluated
  graderId: string            // The grader used for this evaluation

  // Evaluation output
  verdict: Verdict            // "pass", "fail", or "error"
  reason: string              // Explanation from the LLM judge; may be empty string on "error"
}
```

**Constraints:**
- The combination `(experimentId, datasetItemId, graderId)` is unique — there is exactly one cell per item × grader pair per experiment run.
- `verdict` is always one of the three literal values; never null or undefined.
- `reason` is a string; empty string `""` is acceptable when the evaluation errored and no reason was produced.
- Cells in `"error"` state are shown explicitly in the results table (per RunFailurePartial) — they are stored, not discarded.
- On `"error"` verdict, the experiment is marked `"complete"` if at least one cell succeeded, or `"failed"` if all cells errored (per RunFailurePartial / RunFailureTotal).
- The CSV export represents `"error"` cells as the literal string `"error"` in the verdict column (per ExperimentCSVExport).

**Relationships:**
- Belongs to exactly one `Experiment` (many-to-one). Cascade-deleted when the `Experiment` is deleted.
- References one `DatasetItem` and one `Grader` by ID (informational only — these are not cascade-deleted when their referent is deleted; deletion of grader or dataset cascades to the `Experiment` level first, removing all cells transitively).

## Relationship Summary

```
Dataset ──< Attribute          (one-to-many, cascade delete)
Dataset ──< DatasetItem        (one-to-many, cascade delete)
Dataset ──< Experiment         (one-to-many via datasetId, cascade delete)

Grader  ──< Experiment         (many-to-many via graderIds[], cascade delete on grader deletion)

Experiment ──< ExperimentResultCell  (one-to-many, cascade delete)
ExperimentResultCell >── DatasetItem (many-to-one, reference only — no cascade from item side)
ExperimentResultCell >── Grader      (many-to-one, reference only — no cascade from grader side)
```

---

## Cascade Delete Rules (summary)

| Entity deleted | Also deletes |
|---|---|
| `Dataset` | All its `Attribute`s, all its `DatasetItem`s, all `Experiment`s referencing it, all `ExperimentResultCell`s of those experiments |
| `Grader` | All `Experiment`s referencing it (in `graderIds`), all `ExperimentResultCell`s of those experiments |
| `Experiment` | All its `ExperimentResultCell`s |
| `DatasetItem` | Nothing (item deletion does not delete experiments or cells — cells may become stale references, but item deletion is an isolated operation within a live-reference model) |
| `Attribute` (via schema removal) | The corresponding key is dropped from all `DatasetItem.values` in the dataset; no `Experiment` or cell is deleted |

## System-Generated vs. User-Provided Fields

| Entity | System-generated | User-provided |
|---|---|---|
| `Dataset` | `id` | `name` |
| `Attribute` | `id`, `insertionOrder`, `isBuiltIn` | `name` (auto-lowercased) |
| `DatasetItem` | `id`, `insertionOrder` | `values` |
| `Grader` | `id` | `name`, `description`, `rubric` |
| `Experiment` | `id`, `status`, `results` | `name`, `datasetId`, `graderIds` |
| `ExperimentResultCell` | `id`, `verdict`, `reason` | *(none — fully system-produced)* |

## Key Invariants

1. **Schema conformance at all times:** For every `DatasetItem` in a `Dataset`, `Object.keys(item.values)` equals exactly the set of `attribute.name` values in `dataset.schema` (per ItemSchemaConformance).
2. **Built-in immutability:** No operation may remove `"input"` or `"expected_output"` from any dataset's schema (per BuiltInProtection).
3. **Name uniqueness:** Dataset names are unique across all datasets. Grader names are unique across all graders. Attribute names are unique within a dataset's schema.
4. **Non-empty selection for experiments:** A dataset with `items.length === 0` is never available for selection when creating an experiment (per Resolved Decision #6).
5. **At-most-one running experiment:** At most one `Experiment` has `status === "running"` at any time (per ConcurrentRun).
6. **Cell uniqueness:** Within an experiment, the tuple `(datasetItemId, graderId)` uniquely identifies exactly one `ExperimentResultCell`.
7. **Attribute order stability:** `insertionOrder` on `Attribute` is immutable after creation. Built-ins occupy positions 0 and 1.
