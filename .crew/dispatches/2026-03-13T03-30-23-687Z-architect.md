## Design: CSV Import & Export (Dataset Items + Experiment Results)

---

### Goal

When this feature is complete, a user can:
- download a blank CSV template pre-formatted to any dataset's current schema, so they know exactly what columns to fill in before importing
- import multiple dataset items at once by uploading a conforming CSV file
- export all items in a dataset as a CSV file
- export the full results table of any completed experiment as a CSV file

---

### Constraints

- Attribute names are always lowercased and unique within a dataset; `input` and `expected_output` are always present and always first
- All attribute values are strings — no numeric or structured types
- State is in-memory; no persistence to a database is required
- Existing behaviors (ItemCreate, ItemSchemaConformance, BuiltInProtection) must remain fully intact after import
- Import must respect the same validation rules as single-item creation
- Experiment CSV export is only meaningful when the experiment has results; the "Out of Scope" clause in the existing spec explicitly excluded export — this work removes that exclusion for these specific cases only
- This is a single-user, single-session tool — no concurrency concerns on file operations

---

### Approach A: Strict Column Matching

**How it works:** The CSV header row must exactly match the dataset's current schema (same columns, same names, same count). Any deviation is rejected wholesale before any row is processed. The user must fix the file and re-upload.

**Trade-offs:**
- ✅ Predictable — the schema is the contract; ambiguity is impossible
- ✅ Simple to communicate to users ("your file must have exactly these columns")
- ⚠️ Unforgiving — a file with one extra column or a typo fails entirely, even if the data is otherwise valid
- ⚠️ Brittle when schema evolves — a file generated before an attribute was added cannot be re-used without manual editing

**Complexity:** Low
**Scope of change:** Narrow — dataset items section only

---

### Approach B: Lenient Column Matching with Unknown-Column Discard

**How it works:** The CSV header row must contain at minimum `input` and `expected_output`. Any column present in both the file and the schema is imported. Columns in the file that are not in the schema are silently discarded. Columns in the schema that are absent from the file are filled with empty string. Unknown columns trigger a non-blocking warning surfaced after import (e.g. "3 unrecognized columns were ignored").

**Trade-offs:**
- ✅ Tolerant of partial files or files generated from a slightly different schema version
- ✅ Allows bulk `input`/`expected_output` import without worrying about custom attributes
- ⚠️ Silent discard of unknown columns can mask user mistakes (typos in column names go unnoticed)
- ⚠️ Warning text must be clear enough to surface silently-discarded data; easy to miss in a busy UI

**Complexity:** Medium
**Scope of change:** Narrow — dataset items section only

---

### Recommendation

**Approach A** for import; the system is explicitly a structured eval harness, not a general-purpose data tool. Users are expected to work from the template. Strict matching makes the contract obvious and removes a class of subtle data bugs (silently discarded columns). The template download feature exists precisely to eliminate friction around "what columns do I need?" — so Approach B's leniency buys little in exchange for its ambiguity.

For **export** (both dataset items and experiment results), there are no trade-off decisions to surface — the system owns the data and the column order fully. The spec below treats export as deterministic.

---

### Specification

#### Behaviors — Dataset Items

**DatasetCSVTemplateDownload:** When the user requests a CSV template for a dataset, they receive a file containing exactly one row — the header row — with columns in the same order as the dataset's schema: `input` and `expected_output` first, followed by any custom attributes in their insertion order. The file contains no data rows. The filename communicates which dataset it belongs to.

**DatasetCSVImport:** When the user selects a CSV file and confirms the import, the system reads every data row and creates a new dataset item for each one. The items appear in the dataset immediately after the operation completes, appended after any existing items in the order they appeared in the file. No existing items are modified or removed.

**DatasetCSVImportPreview:** Before committing an import, the system shows the user a summary of what will be imported: the number of valid rows found and any rows that will be skipped, with a short reason per skipped row. The user must explicitly confirm before items are created.

**DatasetCSVExport:** When the user requests a CSV export of a dataset's items, they receive a file where the first row is the header (same column order as the schema) and each subsequent row is one item. The file contains all items in insertion order. The filename communicates which dataset it belongs to.

#### Behaviors — Experiment Results

**ExperimentCSVExport:** When the user requests a CSV export of a completed experiment's results, they receive a file where the first row is the header and each subsequent row is one dataset item's results across all graders. The action is only available when the experiment has at least one result. The filename communicates which experiment it belongs to.

---

#### Contracts — Dataset CSV Template

**Template file (download)**
- Format: UTF-8 encoded CSV
- Exactly one row: the header row
- Columns: all attribute names in schema order (`input`, `expected_output`, then custom attributes)
- No BOM, no trailing newline required — consistent with the export format

---

#### Contracts — Dataset Items CSV Import

**Import file (user provides)**
- Format: UTF-8 encoded CSV
- First row: header row containing column names
- Remaining rows: one item per row
- Required columns: `input` and `expected_output` must be present (case-insensitive match — the system normalises to lowercase before comparison)
- Columns matching schema attributes: their values are imported as-is
- Columns not in the schema: treated as unrecognised (see Error Cases)
- Empty cells: treated as empty-string values (same as manual item creation)
- Maximum rows per import: unbounded (no enforced cap for this phase)

**Import preview (system surfaces before commit)**
- `valid_row_count` — integer, number of rows that will be created as items
- `skipped_rows` — list of `{ row_number, reason }` for any rows that will not be imported
- The user must confirm or cancel before any items are created

---

#### Contracts — Dataset Items CSV Export

**Export file (user receives)**
- Format: UTF-8 encoded CSV
- Row 1: header — all attribute names in schema order (`input`, `expected_output`, then custom attributes in insertion order)
- Rows 2–N: one item per row, in insertion order; values are the raw string values stored for each attribute; empty-string values appear as empty cells
- Filename: incorporates the dataset name; exact format left to the implementor

---

#### Contracts — Experiment Results CSV Export

**Export file (user receives)**
- Format: UTF-8 encoded CSV
- Row 1: header
  - First columns: all dataset schema attributes in schema order (`input`, `expected_output`, then custom attributes)
  - Remaining columns: one pair per grader — `{grader_name}_verdict` and `{grader_name}_reason` — in the order graders appear in the experiment; if a grader name contains characters that would be ambiguous in a column header (e.g. commas, quotes), they are escaped per standard CSV quoting rules
- Rows 2–N: one dataset item per row; cells in error state (from `RunFailurePartial`) contain a designated error sentinel value rather than a verdict/reason pair so the user can identify failed cells in the file
- All cells are plain string values; verdicts are the literal strings `pass` or `fail`; the error sentinel is the literal string `error`
- Filename: incorporates the experiment name; exact format left to the implementor

---

#### Error Cases

**ImportNotCSV:** The user provides a file that is not a valid CSV (unreadable, binary, malformed structure) → the import is rejected before the preview is shown and the user sees a message indicating the file could not be parsed as CSV.

**ImportMissingBuiltIns:** The CSV header does not contain `input` or `expected_output` (or both) → the import is rejected before the preview is shown and the user sees a message listing the missing required columns.

**ImportUnknownColumns:** The CSV header contains column names that are not in the dataset's schema → the import is rejected before the preview is shown. The user sees a message listing the unrecognised column names and is told the file must match the dataset's schema exactly. They are reminded they can download the template.

**ImportEmptyFile:** The CSV contains no data rows (only a header, or is completely empty) → the import is rejected and the user sees a message indicating no items were found in the file.

**ImportAllRowsSkipped:** The CSV has data rows, but all of them are skipped due to row-level errors (see `ImportRowBlankBuiltIns`) → the import confirmation is blocked and the user sees a message that no valid rows remain.

**ImportRowBlankBuiltIns:** A data row has an empty value for `input` or `expected_output` → that row is flagged in the preview as skipped, with a reason indicating which built-in attribute is blank. The user sees this before confirming. Other valid rows in the same file may still be imported.

**ImportCancelled:** The user dismisses the preview without confirming → no items are created and the dataset is unchanged.

**ExportEmptyDataset:** The user requests a CSV export of a dataset that has no items → the system still produces a valid file containing only the header row. The user receives the file (a header-only CSV is a valid, usable template equivalent).

**ExportNoResults:** The user attempts to export experiment results when the experiment has no results yet → the export action is unavailable (visually disabled or absent) with a clear indication that results must exist before export is possible.

**ExportPartialResults:** The experiment has results but some cells are in error state (`RunFailurePartial`) → the export proceeds normally; error cells contain the `error` sentinel value. The filename or a brief notice makes clear that some cells contain errors.

---

#### Non-Functional Requirements

- **Encoding:** All CSV files produced by the system are UTF-8 encoded without BOM. All CSV files accepted by the system must be UTF-8 or ASCII; files in other encodings are rejected with a clear message.
- **CSV quoting:** All CSV output must correctly quote cells that contain commas, double-quotes, or newlines, per RFC 4180. Import must tolerate RFC 4180-conforming quoted cells from the user's file.
- **Filename safety:** Generated filenames must not contain characters that are invalid on common operating systems (e.g. `/`, `\`, `:`, `*`). Any such characters present in a dataset or experiment name are replaced or stripped in the filename.
- **Availability gate:** The experiment CSV export action is only surfaced when the experiment's status is `complete` or partially failed (i.e., at least one result cell exists). It is absent or disabled for experiments in `queued` or `running` status.

---

### Out of Scope

- JSON import or export of any kind
- Importing or updating existing items (import is append-only; existing items are never modified or deduplicated by import)
- Merging or replacing a dataset's items wholesale via CSV upload
- Importing a dataset's schema from CSV (schema management remains manual)
- Streaming large CSV files progressively — the full file is read before the preview is shown
- CSV import for graders or experiments
- Any server-side file storage — files are generated on demand and delivered directly to the user; nothing is persisted to disk or a database

---

### Open Questions

1. **Import row limit:** Should a soft warning appear when the import file exceeds a certain row count (e.g., 1,000 rows), even if no hard cap is enforced? The current in-memory model has no explicit item count limit, but very large imports may degrade UI responsiveness.
2. **Header case sensitivity on import:** The spec says column names are matched case-insensitively and normalised. Should the preview confirm to the user which columns were matched (showing the normalised names), or is silent normalisation sufficient?
3. **Experiment export for partial failures:** Is `error` the right sentinel string, or should it be something like `(error)` or an empty cell? The implementor should decide consistently with how error state is already displayed in the results table UI.
4. **Template vs. export naming:** Should the CSV template download and the CSV items export produce identically-structured files (they would share the same header)? If yes, the template is simply an export of an empty dataset — this could simplify implementation but is left for the executor to decide.
