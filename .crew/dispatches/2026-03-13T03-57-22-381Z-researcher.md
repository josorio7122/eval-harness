## Research: Lightweight Job Queue / Background Processing for Node.js Prototypes (Scalable Design)

### Answer

**p-queue (v9.1.0+)** is the right starting point for a prototype needing "simple but scalable" architecture. It's an in-process, promise-based queue with built-in concurrency control via a single `concurrency` option. For your two-level queue pattern (serial experiments + concurrent LLM calls), p-queue can handle both layers, but you'll need to **layer two separate queues** to avoid serializing at the wrong level. This design migrates cleanly to BullMQ (Redis-backed) later without rewriting core logic.

---

### Key Findings

1. **p-queue is production-ready but feature-complete** — Sindre Sorhus explicitly notes "the project is feature complete. We are happy to review pull requests, but we don't plan any further development." Latest version is **v9.1.0**. It's widely used (16M+ weekly downloads) and actively maintained for bug fixes. (source: https://npmjs.com/package/p-queue)

2. **Two-level queue pattern is a well-documented architecture** — The Medium article "Node.js Priority Queues with Abortable Jobs" and "Stop Serializing Everything: 10 Node Patterns" both recommend isolating queues to prevent head-of-line stalls. You need separate queues for different concurrency constraints, not a single queue with mixed rules. (sources: https://medium.com/@2nick2patel2/..., https://medium.com/@Praxen/...)

3. **p-queue supports both serial queuing and rate limiting** — You can use `concurrency: 1` to make experiments run one at a time, and a nested `concurrency: N` queue inside each experiment for parallel LLM calls. p-queue also supports `intervalCap` + `interval` for rate-limit enforcement (e.g., "max 30 LLM calls per minute"). The `strict` mode (v9.1.0+) uses sliding-window rate limiting for API compliance. (source: https://github.com/sindresorhus/p-queue/blob/main/readme.md)

4. **Migration to BullMQ is straightforward** — BullMQ's API is similar but uses `Queue` and `Worker` as separate objects (not queue.process()). BullMQ requires external Redis but adds persistence, clustering, and monitoring (via Bull Board). The safest approach: run Bull and BullMQ queues in parallel with different queue names (`prefix` option), then drain old queues. (source: https://oneuptime.com/blog/post/2026-01-21-bullmq-migrate-from-bull/)

5. **Partial failure handling is native** — p-queue returns a promise per task. Wrap in `.catch()` to handle individual cell failures without crashing the experiment. BullMQ has built-in dead-letter queues (DLQ) if you need persistent failed-job tracking. (source: https://oneuptime.com/blog/post/2026-01-21-bullmq-dead-letter-queue/)

6. **Status tracking requires application logic** — Neither p-queue nor BullMQ store status in the queue itself. You'll need an external source (database, in-memory map) to track "queued → running → complete/failed" states for each experiment. BullMQ exposes job metadata; p-queue requires you to store it separately. (source: https://docs.bullmq.io/guide/jobs)

### Recommended Architecture Pattern

**Three-tier structure:**

```
┌─────────────────────────────────────────┐
│ HTTP API (Hono)                         │
│ POST /experiments/:id/run                │
└────────────┬────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│ Experiment Queue (p-queue)              │
│ concurrency: 1                          │
│ → runs one experiment at a time         │
└────────────┬────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│ Per-Experiment Evaluation Queue         │
│ (nested p-queue inside task)            │
│ concurrency: 5 (or to match API limits) │
│ intervalCap: 30, interval: 60000        │
│ → rate-limit LLM calls                  │
└────────────┬────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│ LLM API Calls (fetch, openai, etc.)     │
│ Each cell is independent; failures      │
│ don't block other cells in same expt.   │
└─────────────────────────────────────────┘
```

**Code shape (TypeScript):**

```typescript
// 1. Global experiment queue (serial)
const experimentQueue = new PQueue({ concurrency: 1 });

// 2. Per-request handler
app.post('/experiments/:id/run', async (req, res) => {
  const experimentId = req.params.id;
  
  // Enqueue the entire experiment
  // Returns immediately; processing happens async
  experimentQueue.add(async () => {
    await runExperiment(experimentId);
  });
  
  res.json({ status: 'queued', experimentId });
});

// 3. Inside runExperiment: nested queue for evaluations
async function runExperiment(experimentId: string) {
  // Update status in DB
  await db.experiments.update(experimentId, { status: 'running' });
  
  // Create evaluation queue scoped to this experiment
  const evalQueue = new PQueue({
    concurrency: 5,
    intervalCap: 30,
    interval: 60000, // 30 calls per 60 seconds
  });
  
  // Get all cells (items × graders)
  const cells = await getExperimentCells(experimentId);
  
  // Add all evaluations to the scoped queue
  const evalPromises = cells.map(cell =>
    evalQueue.add(
      async () => {
        try {
          const result = await callLLM(cell.prompt, cell.model);
          return { cellId: cell.id, result, status: 'success' };
        } catch (error) {
          // Partial failure: log, don't throw
          return { cellId: cell.id, error: error.message, status: 'failed' };
        }
      },
      { priority: cell.priority ?? 0 } // Optional: prioritize certain cells
    )
  );
  
  // Wait for all evaluations (some may fail)
  const results = await Promise.allSettled(evalPromises);
  
  // Collect results and update DB
  const successCount = results.filter(r => r.status === 'fulfilled').length;
  const failureCount = results.length - successCount;
  
  await db.experiments.update(experimentId, {
    status: successCount === results.length ? 'complete' : 'partial',
    results: { successCount, failureCount, cells: results },
  });
}
```

### How to Swap to BullMQ Later

**Path to production (no rewrite of core logic):**

1. **Keep experiment/eval queue split** — Same two-tier pattern, but use BullMQ queues instead
2. **Separate Queue and Worker** — BullMQ decouples them (unlike p-queue)
3. **Redis requirement** — Requires external Redis with AOF persistence and `maxmemory-policy: noeviction`
4. **Migration strategy** — Run both p-queue and BullMQ in parallel (use different queue names or a feature flag)

**Example BullMQ equivalent:**

```typescript
import { Queue, Worker } from 'bullmq';
import { Redis } from 'ioredis';

const connection = new Redis({
  host: process.env.REDIS_HOST,
  port: parseInt(process.env.REDIS_PORT),
});

// Experiment queue (serial)
const experimentQueue = new Queue('experiments', { connection });

const experimentWorker = new Worker('experiments', async (job) => {
  await runExperiment(job.data.experimentId);
}, { connection });

// Inside runExperiment: evaluation queue
async function runExperiment(experimentId: string) {
  const evalQueue = new Queue(`eval-${experimentId}`, { connection });
  
  const cells = await getExperimentCells(experimentId);
  
  // Add all cells as BullMQ jobs
  await evalQueue.addBulk(
    cells.map(cell => ({
      name: `eval-${cell.id}`,
      data: { cellId: cell.id, prompt: cell.prompt },
      opts: { priority: cell.priority ?? 0 },
    }))
  );
  
  // Worker processes them with concurrency limit
  const evalWorker = new Worker(`eval-${experimentId}`, 
    async (job) => {
      const result = await callLLM(job.data.prompt);
      return { cellId: job.data.cellId, result };
    },
    { 
      connection,
      concurrency: 5,
      settings: {
        // BullMQ equivalent of intervalCap
        lockRenewTime: 30000,
        maxStalledCount: 2,
      }
    }
  );
  
  // Wait for queue to drain
  await evalQueue.waitForCompletion(new QueueScheduler(evalQueue).close());
}
```

### Why This Pattern Scales

| Aspect | p-queue (prototype) | BullMQ (production) |
|--------|-------------------|-------------------|
| **Storage** | In-memory (lost on restart) | Redis (persistent) |
| **Multi-process** | Single process only | Multiple workers/machines |
| **Monitoring** | Manual (events + logging) | Bull Board dashboard |
| **Rate limiting** | Supported natively | Supported (with jobOptions) |
| **Partial failure** | Manual catch() | Dead-letter queues + retry |
| **Setup complexity** | ~20 lines | ~100 lines (+ Redis) |

**For a prototype:** p-queue is 10x simpler. Ship fast, monitor with logging.

**When to migrate:** 
- Experiments take >30 min (need persistence across restarts)
- Multiple API servers (need shared queue)
- Thousands of evaluations (need external worker pool)
- Need job history/dead-letter tracking
