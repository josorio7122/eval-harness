## Design System: Eval Harness

### 1. Domain Exploration

#### Domain Concepts

The world of LLM eval is borrowed from experimental science and software testing. The vocabulary is specific:

1. **Rubric** — a grading criterion, like a judge's scorecard. Binary or scored. Has a prompt, a threshold, a rationale.
2. **Dataset** — a curated corpus of input/expected-output pairs. Like a petri dish: controlled, reproducible.
3. **Experiment** — a run of a model against a dataset under a grader. A batch job with a verdict.
4. **Pass / Fail / Error** — the atomic unit of the whole system. A cell in the results grid. The thing the engineer is hunting for.
5. **Score** — aggregate signal. A ratio that collapses complexity into a number the engineer can act on.
6. **Provenance** — traceability. Why did item #47 fail? The trail from score back to grader rationale to model output.
7. **Threshold** — the line between pass and fail. Configurable, consequential.
8. **Ground truth** — the expected output. The thing being compared against.

#### Color World

Walk into the physical version of this space. It's a lab at 11pm. Overhead lights off, monitors on. Think:

- **Deep slate** — the dominant color of terminal backgrounds and IDE themes developers actually work in
- **Phosphor green** — CRT/oscilloscope green, the color of a passing signal, medical EKG lines
- **Warning amber** — instrument panel amber, the color of "proceed with caution" indicators
- **Error red** — not traffic-light red — the deeper, darker red of a system alert, a medical alarm
- **Steel blue** — the color of Vernier calipers, scientific instruments, measurement tools
- **Off-white** — lab notebook paper, the slight warmth of printed tables
- **Ink black** — printed data, the zero-contrast bottom of a dense table

The palette is **dark mode, cool temperature, desaturated except for semantic signals**. Color is reserved for meaning.

#### Signature Element

The **results grid cell** is the signature of this product. This is the entire reason the tool exists. A developer scans dozens of cells looking for red. The signature design:

A dense, borderless grid where each cell has a **colored left-border accent** (2px, green or red or amber) and a **hit-area that expands on hover** to reveal the grader rationale in an inline tooltip. The cell itself shows only the verdict glyph (✓ / ✗ / !) — no text label, no padding waste. The row label is the dataset item; the column label is the grader name. This is a scoreboard, not a report.

No other dashboard tool has this exact construct. Remove the product name — you'd still know it's an eval harness.

#### Defaults to Reject

| Default | Why it's wrong | What replaces it |
|---|---|---|
| White card grid with soft shadows | Feels like a SaaS dashboard, not a scientific instrument. The warmth and roundness contradicts "I can trust these results." | Dark canvas, borders-only depth, sharp radius (4px). Every surface is a measurement instrument, not a product tile. |
| Color-coded status badges (pill-shaped) | Generic. Used by every issue tracker and CI tool. Loses the density of the results matrix. | Left-border accent on table rows. Color lives at the edge of the row, not inside a badge. Status is communicated by the presence/absence of color, not by label. |
| Blue primary action buttons | Blue is the wrong anchor in a system where **steel blue** will be used for navigation/selection and green/red/amber are the meaningful signals. Blue "actions" fight with blue "selected state." | Near-white action buttons on dark surfaces (`--surface-3` background, `--fg-primary` text, `--border-strong` border). Primary actions are restrained. The results speak louder than the buttons. |

### 2. Design Checkpoint

**Intent:** An ML engineer at midnight, iterating on a stubborn eval. They're not browsing — they're hunting. Scanning the results grid for red cells, clicking through to understand why a grader failed, adjusting the rubric, rerunning. The tool should feel like a precision instrument: high-density, low decoration, every pixel earning its place. The feel is a lab notebook crossed with a terminal — dark, cool, structured, exact.

**Palette:**

The foundation is `zinc` — not blue-gray (too cool), not stone (too warm) — zinc sits at the neutral intersection that reads as "technical" without the clinical chill of pure slate. The accent is a single, restrained **steel-blue** (`hsl(215, 60%, 58%)`) used only for interactive selection and focus. Semantic colors carry the emotional weight.

```
--zinc-950:  hsl(240, 6%, 7%)     ← base canvas
--zinc-900:  hsl(240, 5%, 11%)    ← cards / panels
--zinc-800:  hsl(240, 4%, 16%)    ← elevated surfaces
--zinc-700:  hsl(240, 3%, 22%)    ← borders, dividers
--zinc-600:  hsl(240, 3%, 30%)    ← muted borders
--zinc-400:  hsl(240, 4%, 54%)    ← tertiary text
--zinc-300:  hsl(240, 4%, 66%)    ← secondary text
--zinc-100:  hsl(240, 5%, 90%)    ← primary text

--accent:    hsl(215, 60%, 58%)   ← steel-blue, selection/focus only
--pass:      hsl(142, 52%, 44%)   ← phosphor green, desaturated
--fail:      hsl(0, 60%, 52%)     ← deep red
--error:     hsl(38, 85%, 52%)    ← instrument amber
--neutral:   hsl(240, 3%, 40%)    ← skipped / n/a cells
```

Semantic colors are desaturated from their "pure" values — `hsl(142, 52%, 44%)` is green but not neon. It reads as "pass" without screaming.

**Depth strategy:** Borders-only. A pure, uncompromised decision. Shadows are perception of physical light; this tool is not physical. The grid, the panels, the inputs — all separated by `0.5px–1px solid rgba(255,255,255,0.07–0.12)`. The tool reads like a terminal with structure, not a dashboard with lift.

**Surfaces:**
```
Level 0 (canvas):   hsl(240, 6%, 7%)     ← the app background
Level 1 (panels):   hsl(240, 5%, 11%)    ← main content areas, cards
Level 2 (raised):   hsl(240, 4%, 16%)    ← inputs, dropdowns, popovers
Level 3 (overlay):  hsl(240, 3%, 20%)    ← modal backgrounds, top sheets
```
Each jump is 4–5 percentage points of lightness. Whisper-quiet.

**Typography:** `Inter` via system font stack — works directly with shadcn/ui defaults. Dense at 13px base, not 16px. The tool is information-dense; body copy is a rarity.

**Spacing base unit:** 4px.

### 3. Token Architecture

#### Foreground Hierarchy

```
--fg-primary:     hsl(240, 5%, 90%)   ← default text (zinc-100)
--fg-secondary:   hsl(240, 4%, 66%)   ← supporting text (zinc-300)
--fg-tertiary:    hsl(240, 4%, 54%)   ← metadata, timestamps (zinc-400)
--fg-muted:       hsl(240, 3%, 38%)   ← disabled, placeholder (zinc-500ish)
--fg-inverted:    hsl(240, 6%, 7%)    ← text on light surfaces (zinc-950)
```

#### Background / Surface Levels

```
--bg-base:        hsl(240, 6%, 7%)    ← canvas
--bg-surface-1:   hsl(240, 5%, 11%)   ← panels, cards
--bg-surface-2:   hsl(240, 4%, 16%)   ← inputs, code blocks, table headers
--bg-surface-3:   hsl(240, 3%, 20%)   ← dropdowns, popovers, modals
--bg-inset:       hsl(240, 6%, 5%)    ← recessed areas, pre/code, empty states
```

#### Border Scale

```
--border-subtle:  rgba(255,255,255,0.05)   ← faintest separation (table rows)
--border-default: rgba(255,255,255,0.08)   ← standard card/panel borders
--border-strong:  rgba(255,255,255,0.12)   ← interactive element borders
--border-focus:   hsl(215, 60%, 58%)       ← focus ring = accent color
```

#### Brand Accent

```
--accent:         hsl(215, 60%, 58%)       ← steel-blue
--accent-subtle:  hsla(215, 60%, 58%, 0.12) ← accent tint for selection bg
--accent-strong:  hsl(215, 60%, 70%)       ← hover on accent elements
```

#### Semantic Colors (the critical ones)

```
/* Pass — phosphor green, desaturated for credibility */
--pass:           hsl(142, 52%, 44%)
--pass-subtle:    hsla(142, 52%, 44%, 0.12)   ← cell background tint
--pass-fg:        hsl(142, 52%, 70%)           ← text on dark bg

/* Fail — deep red, not alarming red */
--fail:           hsl(0, 60%, 52%)
--fail-subtle:    hsla(0, 60%, 52%, 0.12)
--fail-fg:        hsl(0, 60%, 75%)

/* Error — instrument amber (grader itself failed) */
--error:          hsl(38, 85%, 52%)
--error-subtle:   hsla(38, 85%, 52%, 0.12)
--error-fg:       hsl(38, 85%, 75%)

/* Neutral — skipped, n/a, pending */
--neutral:        hsl(240, 3%, 40%)
--neutral-subtle: hsla(240, 3%, 40%, 0.15)
--neutral-fg:     hsl(240, 4%, 60%)
```

**Design rule:** Semantic colors appear only in three places — the result cell (left-border accent + optional background tint), the status indicator in the row/column summary, and the aggregate stat. Nowhere else. A passing experiment does not paint the header green.

### 4. Typography

#### Font Stack

```
--font-sans:   'Inter', 'Inter var', system-ui, -apple-system, sans-serif
--font-mono:   'JetBrains Mono', 'Fira Code', 'SF Mono', 'Cascadia Code', Consolas, monospace
```

Rationale: Inter is the shadcn/ui default and renders crisply at 12–13px. JetBrains Mono is the preferred monospace for developers who write code daily — it has ligatures for `!=`, `>=`, `->` that appear in rubric logic displays. Falls back gracefully.

#### Size Hierarchy

| Role | Size | Weight | Letter-spacing | Font |
|---|---|---|---|---|
| Page title | 16px | 600 | -0.01em | sans |
| Section heading | 13px | 600 | 0.01em | sans, uppercase |
| Body / description | 13px | 400 | 0 | sans |
| Table header | 11px | 600 | 0.05em | sans, uppercase |
| Table cell | 13px | 400 | 0 | sans |
| Data / scores | 13px | 500 | 0 | mono, tabular-nums |
| Stat number | 24px | 600 | -0.02em | mono, tabular-nums |
| Stat label | 11px | 500 | 0.04em | sans, uppercase |
| Code / prompt text | 12px | 400 | 0 | mono |
| Badge / pill label | 11px | 500 | 0.03em | sans |
| Timestamp / meta | 11px | 400 | 0 | sans |

**Critical rule:** All numeric data (scores, counts, pass rates, timestamps) uses `font-variant-numeric: tabular-nums` and the mono stack. Numbers in a column must align on their decimal points. A column of pass rates that wiggles left-right is a precision failure.

### 5. Spacing

**Base unit: 4px.** All spacing is a multiple.

```
--space-1:   4px    ← icon gap, tight inline pairs
--space-2:   8px    ← within control elements (button padding-y, icon-label gap)
--space-3:   12px   ← standard padding within cards, cell horizontal padding
--space-4:   16px   ← section padding, gap between form fields
--space-5:   20px   ← card gap in lists
--space-6:   24px   ← major section separation
--space-8:   32px   ← top-level page sections
--space-10:  40px   ← maximum section gap
```

**Table cell padding:** `8px 12px` (vertical space-2, horizontal space-3). Dense but not airless.  
**Card padding:** `16px` all sides (space-4).  
**Input height:** `32px` with `6px 10px` internal padding — tight, terminal-like, not form-field generous.  
**Button height:** `32px` for default, `28px` for compact (table inline), `36px` for primary CTA.

### 6. Depth Strategy

**Borders-only. No exceptions.**

Why: This tool is used in dark contexts (dark OS, dark IDE, dark browser). Shadows rely on simulated light sources — they read less clearly against dark canvases and require opacity tuning that conflicts with the zinc-on-zinc elevation system. Borders communicate structure without implying physicality. The tool is not a set of floating cards; it is a structured workspace. Flat structural boundaries fit the "precision instrument" mental model. Every competing tool in this space (Langsmith, Braintrust, Phoenix) either uses a light mode or shadow-heavy dark mode — borders-only is a genuine differentiator that reinforces the terminal aesthetic.

The border discipline:
- `0.5px` borders within data structures (table row dividers, cell separators)
- `1px` borders around panels, cards, interactive elements
- **Never** `border: 1px solid #333` — always `rgba(255,255,255, X)` alpha for subtlety

### 7. Component Patterns

#### Navigation

A two-tier structure: a **top bar** (24px tall, `--bg-surface-1`, right-aligned experiment runner status and user meta) + a **left sidebar** (220px, same background as canvas — `--bg-base` — with a `1px solid --border-default` right edge). The sidebar contains the three primary sections: Datasets, Graders, Experiments. No icons-only collapsed state; this is a tool, not a product with brand space constraints.

Active section: a 2px left-border in `--accent` on the nav item, no filled background. Hover: `--bg-surface-1` background fill at 100% width.

Section headings in the sidebar (`DATASETS`, `GRADERS`, `EXPERIMENTS`) are 10px, 600 weight, uppercase, `--fg-tertiary` — they are orientation markers, not interactive targets.

---

#### Dataset List

**Design intent:** A dense list, not a card grid. Each dataset is a table row: name (primary), item count + schema field count (secondary, monospace), last-modified timestamp (tertiary), and an inline action affordance on hover.

The list uses a `1px solid --border-subtle` row separator. No zebra striping — that's a spreadsheet convention. Hover state: `--bg-surface-1` background fill across the full row width.

A dataset in "incomplete schema" state (no fields defined yet) shows an amber left-border accent — same token as `--error-fg`. Not alarming, just advisory.

**States:**
- **Loading:** Three rows of shimmer. The shimmer is a horizontal gradient scan using `--bg-surface-2` as the highlight, `--bg-surface-1` as the base. No spinner. The shimmer should match row height exactly.
- **Empty:** A `--bg-inset` centered panel with a line-drawing style icon (Lucide `Database`, `--fg-muted` color), a single short line of `--fg-secondary` text ("No datasets yet"), and a single action button. No illustration, no cheerful copy.
- **Error:** Same panel structure. Lucide `AlertCircle` in `--error-fg`. Message in `--fg-secondary`. A retry link.
- **Populated:** The list. Clicking a row navigates to Dataset Detail.

---

#### Dataset Detail (Schema + Items)

**Design intent:** A two-panel view with a persistent vertical split. Left panel (30%): schema fields as a compact definition list — field name (mono, `--fg-primary`), type badge (11px, `--bg-surface-2` background, `--fg-secondary`), required indicator. Right panel (70%): the items table.

The items table is the workhorse. Columns are schema field names (truncated, with tooltip on overflow). Values are displayed in mono at 12px. Long text values truncate to 2 lines with a `...` overflow indicator and expand on row click to a side sheet.

The schema panel and items panel share the same `--bg-surface-1` background. They are separated by a `1px solid --border-default` vertical divider. No header, no card wrap — the panels ARE the surface.

**States:**
- **Schema empty:** A faint dashed border around the schema panel area, `--fg-muted` text "Define schema fields" in the center. A `+` button at the bottom of the panel.
- **Items empty (schema exists):** The items panel shows the `--bg-inset` empty state with the column header still visible — this tells the user what structure they're importing into.
- **Loading items:** Row shimmer. Schema remains visible immediately (cached).

---

#### Grader List

**Design intent:** Similar to dataset list but each row shows the grader name, type badge (`LLM` / `Exact Match` / `Regex` — these are different instruments), and a short truncated description of the rubric criterion.

The `LLM` type badge is `--accent-subtle` background with `--accent` text — it is the interesting, expensive one. `Exact Match` and `Regex` use `--bg-surface-2` / `--fg-secondary`.

**States:** Same loading/empty/error pattern as dataset list.

---

#### Grader Detail (Rubric Editor)

**Design intent:** This is where the rubric criterion lives. It is a form, but it should feel like editing a document in a structured editor, not filling out a SaaS settings form.

The rubric prompt occupies most of the vertical space — a `--bg-inset` textarea, mono font, 12px, with a subtle `1px solid --border-default` border that becomes `--border-focus` (accent blue) on focus. No resize handle; auto-grows to content. The textarea is not labeled "System Prompt" — it is labeled "Grading criterion" in the 11px uppercase section heading style.

Below the prompt: threshold input (a number `0.0–1.0` with tab-stops), pass condition selector (custom dropdown), model selector (custom dropdown — not native `<select>`).

A live preview panel sits to the right (split layout): shows the grader's last run outputs for 3–5 sample items. This keeps the rubric connected to its real outputs during editing.

**States:**
- **Loading grader:** The textarea shows a `--bg-surface-2` placeholder rectangle, same height as expected content.
- **Unsaved changes:** A `1px solid --error` left-border appears on the entire form panel — a thin signal that this instrument is uncalibrated. An "Unsaved" label in `--error-fg` replaces the normal status in the panel header.

---

#### Experiment List

**Design intent:** The experiment list needs to communicate run status at a glance. Status is shown by a **left-border accent** on each row, not a badge pill:

- Running: `--accent` (steel-blue) — active, in progress
- Completed (pass rate ≥ threshold): `--pass` (green)
- Completed (pass rate < threshold): `--fail` (red)
- Failed/errored: `--error` (amber)
- Pending/queued: `--neutral` (gray)

The row itself shows: experiment name, dataset name (secondary), grader name (secondary), item count × grader count (mono, tertiary), timestamp, and a pass-rate aggregate number (`67% · 80/120` in mono, colored to match status).

Progress bar for running experiments: a thin `2px` horizontal bar at the bottom of the row (not in a separate column) — fills left to right as items complete, color `--accent`, background `--bg-surface-2`. It replaces the pass-rate display until complete.

**States:**
- **Loading:** Row shimmer with a consistent `--neutral` left-border placeholder.
- **Empty:** `--bg-inset` panel, Lucide `FlaskConical`, "No experiments yet", action button.
- **Running (live):** Rows with `--accent` left-border and the inline progress bar update in real-time. New rows insert at the top with a brief entry animation.

---

#### Results Table (The Core View)

This is the product. It is a matrix: rows are dataset items (identified by a key field from the schema), columns are graders applied to this experiment. Each cell is a verdict.

**Cell anatomy:** The cell is `44px` tall in default state. It contains:
- A **2px left-border** in the semantic color (`--pass`, `--fail`, `--error`, or `--neutral`)
- An **optional 8% opacity background tint** using `--pass-subtle` / `--fail-subtle` / `--error-subtle` — only on fail and error cells, not pass (pass is the expected state; fail and error demand attention)
- A **verdict glyph** centered: `✓` in `--pass-fg`, `✗` in `--fail-fg`, `!` in `--error-fg`, `—` in `--neutral-fg`. The glyph is 14px, weight 600, mono-rendered for alignment.
- **No text label.** Pass/Fail/Error is communicated by glyph + color. Text labels would make the table too wide to scan.

**Hover state:** On cell hover, the row height stays fixed but the cell expands via an overlay popover anchored to the cell position. The popover (150-250ms transition) shows:
- The item key + grader name (header)
- The model output (scrollable, mono, 12px, capped at 8 lines)
- The grader rationale (the "why" — `--fg-secondary`)
- The raw score if numeric

The popover uses `--bg-surface-3` background, `1px solid --border-strong` border. It overlays the rows below — it does not push them.

**Column headers:** Grader names. 11px, uppercase, 600 weight, `--fg-tertiary`. Rotated 45° if there are more than 6 graders to prevent column width explosion. This is the only intentional non-horizontal text in the product.

**Row headers:** Dataset item key values. 13px, mono, `--fg-secondary`, truncated at 160px with Lucide `MoreHorizontal` indicator.

**Aggregate row** (pinned at bottom): A sticky summary row with each column's pass rate. `—/—` count in mono + a small inline mini-bar (Tremor `ProgressBar`, 8px height, `--pass` fill, `--bg-surface-2` track, spanning the cell width). This row has `--bg-surface-2` background — one level elevated from the data rows.

**Row sorting:** Rows default sorted by fail count descending — the failures bubble to the top. This is the critical default. An engineer opens results to find what broke; failing rows first means zero scrolling to the problem.

**Selection:** Clicking a row header selects the row (the row gets `--accent-subtle` background wash + `--accent` left-border) and opens a right-side slide panel with full item detail.

**States:**
- **Loading:** A shimmer grid — the row/column header skeleton loads first (gives structure), then cell shimmers fill in.
- **Running (live):** Cells fill in as results arrive. A cell that has not yet received a result shows a `--neutral` subtle pulsing state (opacity 0.4 → 0.7 → 0.4 loop, 1.5s cycle). Not a spinner — a breathe.
- **Empty (no items):** Blank grid with `--bg-inset` fill and empty state copy.
- **Complete with 100% pass:** No special visual flourish. The table is green-bordered and that is the reward.

---

#### Aggregate Stats Area

Positioned above the results table as a fixed-height `--bg-surface-1` banner row (80px). Three stat blocks:

1. **Pass Rate** — the headline number. `24px` mono, `--pass-fg` colored when above threshold, `--fail-fg` when below, `--fg-secondary` when calculating.
2. **Item count × Grader count** — `13px` mono, `--fg-tertiary`. `"120 items × 3 graders = 360 evaluations"`
3. **By-Grader breakdown** — Tremor `BarChart`, constrained to the width of this block, showing each grader's pass rate as a bar. Bar fill uses `--pass` / `--fail` based on whether individual grader rate is above threshold. No axis labels (too dense). Tooltip on hover showing exact value.

The stat banner is separated from the table by a `1px solid --border-default` bottom border. It does not scroll with the table.

**States:**
- **Calculating:** The pass rate shows a `—` placeholder, a Lucide `Loader2` icon spinning at 16px beside it in `--fg-muted`. No percentage shown. The bar chart shows neutral bars.
- **Error:** If any grader errored on more than 20% of items, a `--error` colored warning strip appears below the stat banner: `"Grader X failed on 34 of 120 items — results incomplete"`.

#### Pass/Fail/Error Cell Anatomy (Summary)

This is the atomic design unit of the product. Specify it precisely:

```
Cell container:
  height: 44px (default view)
  padding: 8px 12px
  border-left: 2px solid <semantic-color>
  background: <semantic-subtle> (fail/error only)

Glyph:
  content: ✓ / ✗ / !  / —
  font: mono, 14px, weight 600
  color: <semantic-fg>
  display: flex, align-center, justify-center

Hover trigger:
  cursor: default (not pointer — this is not a link)
  On hover: popover appears after 150ms delay (prevents flash on scan)

Popover:
  background: --bg-surface-3
  border: 1px solid --border-strong
  border-radius: 6px
  padding: 12px
  max-width: 320px
  position: anchored below cell if room, above if near bottom
  z-index: overlay layer
```

#### Progress Indicator for Running Experiments

Two expressions depending on context:

**In experiment list row:** A `2px` bottom-border progress line replacing the pass-rate stat. Color `--accent`. Fills left to right. Background `--bg-surface-2`. Below the row content, not inside it. Shows `"47 / 120"` count in mono, `--fg-tertiary`, left-aligned in the stat column.

**In results table (live run):** Each cell that has completed brightens from the neutral breathing state to its final color in a `200ms` opacity transition. There is no percentage bar in the table header — the table IS the progress indicator.

**In experiment detail header:** A Tremor `ProgressBar` component, `8px` height, `--pass` fill, `--bg-surface-2` track, full-width of the header area. A label: `"47 of 120 evaluated · 14 failed so far"` in 12px, `--fg-secondary`.

---

#### Tab Navigation (within Detail Views)

Dataset detail has two tabs: **Schema** and **Items**. Grader detail has two: **Rubric** and **History**. Experiment detail has: **Results**, **Stats**, **Log**.

Tab style: NOT pill tabs. NOT underline tabs with colored bottoms. Instead: tabs are a row of text buttons (`13px`, `--fg-tertiary` inactive, `--fg-primary` active, weight 500). The active tab has a `1px solid --border-default` bottom border in `--fg-primary` color — a thin underline, same color as the text. The tab row sits on `--bg-surface-1` with a `1px solid --border-default` bottom border separating it from content.

This is a precision instrument. The tab design should feel like selecting a measurement mode on a multimeter — a mode selector, not navigation.

### 8. States — Comprehensive Matrix

| View | Loading | Empty | Error | Populated |
|---|---|---|---|---|
| Dataset list | 3 row shimmers, height-matched | Inset panel, DB icon, short copy, 1 action | Inset panel, alert icon, retry link | Dense table rows, hover reveals actions |
| Dataset detail (schema) | Single block shimmer in left panel | Dashed border panel, `+` button | Alert + retry | Definition list, compact |
| Dataset detail (items) | Row shimmers with column headers visible | Inset panel with column structure shown | Alert in table body | Scrollable table, mono values |
| Grader list | 3 row shimmers | Inset panel, gavel icon, copy, 1 action | Standard error panel | Table rows with type badges |
| Grader detail | Full form shimmer | N/A (form is always in a state) | Error inline in form area | Split form/preview |
| Experiment list | Row shimmers with neutral left-border | Inset panel, FlaskConical icon | Standard error panel | Rows with colored left-borders |
| Results table | Grid shimmer (headers first) | Inset panel centered | Error strip at top + partial data if partial | Dense matrix, fail rows sorted first |
| Aggregate stats | `—` with Loader2 spinner | N/A (calculated from results) | Error strip with grader detail | Three stat blocks + Tremor chart |

### 9. Animation

This is a professional tool, not a consumer product. Animation communicates state change — it does not delight.

**Where Motion (Framer Motion) is used:**

| Element | Transition | Duration | Easing |
|---|---|---|---|
| Results cell hover popover | `opacity: 0 → 1`, `y: 4px → 0` | 150ms | `ease-out` |
| Popover dismiss | `opacity: 1 → 0` | 100ms | `ease-in` |
| Sidebar nav item active | Left-border `width: 0 → 2px` | 150ms | `ease-out` |
| New experiment row insert | `height: 0 → auto`, `opacity: 0 → 1` | 200ms | `ease-out` |
| Result cell fill (live run) | `opacity: 0.4 → 1` | 200ms | `ease-out` |
| Breathing cell (pending) | `opacity: 0.4 ↔ 0.7` | 1500ms loop | `ease-in-out` |
| Slide panel (item detail) | `x: 100% → 0` | 250ms | `ease-out` |
| Slide panel dismiss | `x: 0 → 100%` | 200ms | `ease-in` |
| Tab content switch | `opacity: 0 → 1` | 100ms | `ease-out` |
| Progress bar fill | `width` CSS transition (not Motion) | continuous | linear |
| Shimmer scan | CSS `@keyframes` gradient shift | 1200ms loop | linear |

**What does NOT animate:**
- Table row reorder (too disorienting in a data grid — reorder is instant)
- Page navigation (instant — the tool is not a wizard)
- Form input state changes (instant)
- Status badge changes (instant)
- Modal overlay backdrop (instant `opacity` snap, not fade)

**The guiding rule:** If the animation would still feel right in a terminal emulator, it belongs here. If it would look out of place in `htop`, it doesn't.

### Craft Checks

**Swap test:** Replace the font with `system-ui`, change the accent to generic blue, remove the left-border accents — does it become indistinguishable from a generic dark admin panel? Yes. That's exactly why those three elements exist. They are load-bearing identity.

**Squint test:** Squint at the results table. The fail cells (amber/red tinted backgrounds, red left-borders) should cluster visibly at the top due to the default sort. You should be able to locate problem rows before your eyes focus. If the grid is visually uniform, the semantic coloring is insufficient.

**Signature test:** The five signature appearances: (1) the left-border cell accent in the results matrix, (2) the `2px` progress line at the row bottom (not a bar in a column), (3) the stat row pinned at the table bottom with inline mini-bars, (4) the rubric editor's unsaved-state left-border on the panel, (5) the breathing cell animation for pending results during a live run. None of these come from the shadcn/ui defaults.

**Token test:** Reading the variable names aloud — `--pass-fg`, `--fail-subtle`, `--bg-inset`, `--border-default` — they sound like a scientific instrument's HMI. Not `--primary`, `--success`, `--danger`. The naming itself signals this product's purpose.
